# Exam Platform Requirements

## 1. Project Overview

Build a modern, clean, light-themed web application for generating and conducting MCQ-based tests.

The platform must initially use **CSV files as the database**, while being designed so it can later be switched to **PostgreSQL** through a **settings option in the admin interface**.

The application has two main roles:

1. **Admin**
   - manages question bank
   - creates test sessions
   - generates candidate test links
   - reviews candidate results
   - controls active storage backend

2. **Candidate/User**
   - opens a test link generated by admin
   - fills in personal details
   - takes a timed, randomized MCQ exam
   - submits or ends exam early

---

## 2. Business Goal

The purpose of the website is to help admins manage a question repository and quickly generate test sessions for candidates without needing a full database setup at the start.

The system should support:

- low-friction local operation using CSV files
- future migration to PostgreSQL without major rewrites
- simple administration of questions and tests
- capture of all candidate results and metadata

---

## 3. Core Product Vision

The product should feel:

- **modern**
- **clean**
- **light-themed**
- **interactive**
- **easy to navigate**
- **responsive on desktop and laptop screens**

The admin experience should feel like a lightweight dashboard, while the candidate experience should feel focused and distraction-free.

---

## 4. Functional Requirements

### 4.1 Admin Authentication

The system must provide an admin login flow.

#### Requirements
- Admin must be able to log in using email and password.
- Admin-only pages must be protected.
- Unauthorized users must not access admin dashboard routes.
- For version 1, simple local authentication is acceptable.

---

### 4.2 Question Management

Admins must be able to manage a question bank from the website.

#### Question Fields
Each question should contain:

- unique question ID
- question text
- multiple answer options
- one correct answer
- enabled/disabled status
- created timestamp
- updated timestamp

#### Required Admin Actions
- create a new question
- create multiple questions
- edit/update an existing question
- delete a question
- disable a question without deleting it
- enable a previously disabled question
- view all questions in a list/table/dashboard view

#### Business Rules
- only enabled questions can be used in test generation
- deleted questions should no longer appear in active question lists
- updates should refresh the `updatedAt` timestamp

---

### 4.3 Test Session Creation

Admins must be able to create shareable test sessions.

#### Session Fields
Each session should contain:

- unique session ID
- test title
- number of questions to serve
- time limit in minutes
- created by admin
- created timestamp
- active/inactive status

#### Required Actions
- create a test session
- activate/deactivate a session
- generate a shareable test link using the session ID

#### Test Link Behavior
- generated link should include or resolve using the session ID
- candidate access must validate that the session exists and is active

---

### 4.4 Candidate Entry Flow

When a candidate opens the generated session link, the app must first ask for candidate details before the exam starts.

#### Required Candidate Fields
- full name
- email address
- phone number

#### Required Flow
1. candidate opens test link
2. system validates session
3. candidate enters required personal details
4. candidate clicks **Start Test**
5. exam begins

---

### 4.5 Candidate Exam Flow

The exam experience must be interactive and focused.

#### Required Behavior
- questions must be populated randomly from enabled questions
- questions should appear **one by one**
- the test must support a **timer**
- candidate can move through the exam flow and answer questions
- candidate can **end exam early**
- exam should be submitted automatically if timer expires

#### Exam UI Expectations
- clear question card
- visible progress indicator
- visible timer
- answer selection controls
- next/previous navigation if implemented
- end exam button with confirmation

#### Business Rules
- only enabled questions are eligible
- question order should be randomized per attempt
- question count should respect the session configuration

---

### 4.6 Result Capture

The system must capture candidate attempts and answers.

#### Attempt Data
Each attempt should store:

- attempt ID
- session ID
- candidate name
- candidate email
- candidate phone
- start time
- end time
- status (`in_progress`, `completed`, `timed_out`, `ended_early`)
- score
- total questions
- answered question count

#### Answer Data
Each saved answer should store:

- attempt ID
- question ID
- selected answer
- whether the answer is correct
- answer timestamp
- question order

---

### 4.7 Results Dashboard

Admin must be able to review test performance.

#### Required Features
- list all attempts
- filter attempts by session if needed
- see candidate details
- see score and completion status
- see time-related information

---

### 4.8 Storage Backend Switching

The app must support two storage modes:

1. **CSV** for initial version
2. **PostgreSQL** for future-ready mode

#### Required Behavior
- app logic should depend on a shared storage abstraction/repository interface
- admin settings page should expose a visible switch or selector for storage provider
- if PostgreSQL is not configured, the app should safely remain on CSV
- switching storage must not require rewriting the UI layer

#### Version 1 Expectation
- CSV is the default and active implementation
- PostgreSQL support can initially exist as a scaffold/provider placeholder

---

## 5. Data Model Requirements

### 5.1 Question
- `id: string`
- `questionText: string`
- `options: string[]`
- `correctAnswer: string`
- `isEnabled: boolean`
- `createdAt: string`
- `updatedAt: string`

### 5.2 Test Session
- `sessionId: string`
- `title: string`
- `questionCount: number`
- `timeLimitMinutes: number`
- `createdBy: string`
- `createdAt: string`
- `isActive: boolean`

### 5.3 Attempt
- `attemptId: string`
- `sessionId: string`
- `name: string`
- `email: string`
- `phone: string`
- `startedAt: string`
- `endedAt: string | null`
- `status: in_progress | completed | timed_out | ended_early`
- `score: number | null`
- `totalQuestions: number`
- `answeredQuestions: number`

### 5.4 Attempt Answer
- `attemptId: string`
- `questionId: string`
- `selectedAnswer: string`
- `isCorrect: boolean`
- `answeredAt: string`
- `questionOrder: number`

### 5.5 App Settings
- `storageProvider: csv | postgres`
- `postgresConfigured: boolean`

## 10. Acceptance Criteria

The platform should be considered complete when:

1. admin can log in
2. admin can add, update, delete, enable, and disable questions
3. admin can create a timed session and generate a test link
4. candidate can enter name, email, and phone before starting
5. candidate receives randomized one-by-one MCQ questions
6. candidate sees and is governed by a timer
7. candidate can end exam early
8. exam auto-submits on timeout
9. all results are stored in the active backend
10. admin can view captured attempts and scores
11. admin can control storage provider from settings
12. code structure supports future PostgreSQL migration without major rewrite
